## Audit FIL 2 — BaseController (premier constat)

Très bon candidat pour commencer : on voit immédiatement ce qui relève de l’orchestration (bien), et ce qui dérive un peu.

## ✅ Ce que le contrôleur fait bien

Architecture globalement saine :

```text id="wn8z5z"
BaseController
├── initialise le contexte
├── gère la compétition active
├── injecte les données communes vues
└── fournit quelques garde-fous (requireCompetition)
```

Ça correspond bien à un rôle de BaseController.

---

# Points d’audit

## 1. ⚠️ Double source de vérité compétition active

Ici :

```php
session()->get('competition_id')
CompetitionService::getActive()
ensureActiveCompetition()->where('active',1)
```

On a **3 mécanismes** pour définir "la compétition active".

Problème :

* risque de divergence
* logique métier dispersée
* violation du "une règle = un endroit"

### cible :

```text id="vaf7my"
BaseController ne décide pas la compétition active.
Il la demande à un seul service.
```

Idéal :

```text
CompetitionContextService
└── getActiveCompetition()
```

BaseController orchestre seulement.

👉 Premier refactor prioritaire.

---

## 2. CompetitionService dans un Controller ⚠️

```php
CompetitionService::getActive();
```

Si service statique métier :

* difficile à tester
* cache souvent du global
* tendance "service god object"

Préférable :

* injecter service
* ou repository métier

Audit drapeau jaune.

---

## 3. Duplication suspecte

Ça :

```php
$model = new CompetitionModel();
$competitionModel = model('CompetitionModel');
```

double instanciation.

Une doit disparaître.

Et probablement :

```php
helper('competition');
```

à vérifier :

* vrai utilitaire ?
* ou logique métier cachée dans helper ?

Les helpers dérivent vite.

---

## 4. Data partagées vues dans BaseController ✅ (mais surveiller)

Bien :

```php
competitions
competitions_list
activeCompetition
competitionId
```

Mais :

```php
getCompetitionsWithCount()
```

si KPI / agrégats lourds → pas forcément BaseController.

Question audit :

* est-ce une donnée de layout ?
  ou logique dashboard ?

Si dashboard-only → sortir d’ici.

Très important.

---

## 5. ensureActiveCompetition() semble hors place

Celui-ci :

```php
protected function ensureActiveCompetition()
```

fait :

* requête DB
* décision métier
* redirect HTTP

mélange :

* repository
* business rule
* controller flow

odeur de responsabilité mixte.

Je le marquerais candidat refactor.

---

# 6. Très bonne partie : garde-fous

Très propre :

```php
getCompetitionId()
requireCompetition()
setCompetition()
```

Ça, à garder.

C’est du vrai rôle BaseController.

---

# 7. Petite dérive debug

```php
log_message('debug' ...)
```

OK provisoirement.

Mais si logs structurants :

* passer via service de trace
* ou centraliser

mineur.

---

# Diagnostic synthèse

```text id="y1g15j"
BaseController
```

et sortir :

```text
where(active=1)
fallback rules
session sync logic
```

du controller.

```text id="sdcw7u"
Structure : bonne
Complexité : modérée
Dette technique : localisée
```

Très récupérable.

## Ma recommandation

Avant tout refactor :

1. figer la règle officielle "active competition" (Bible FIL1)
2. supprimer les 3 mécanismes concurrents
3. faire de BaseController un simple orchestrateur

C’est probablement un chantier de 30 lignes, pas plus.
